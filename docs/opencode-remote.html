Remote OpenCode

Point OpenCode at a remote Lemonade server.

OpenCode uses an OpenAI-compatible provider config, so the remote setup is mostly about the right baseURL, a valid API key, and the exact Lemonade model ID.

Config

Use a remote base URL, not localhost.

Your sample config is already structured correctly. For remote use, the important change is replacing the local OpenAI-compatible endpoint with the remote one.

Local example

"options": {
  "baseURL": "http://localhost:8000/api/v1",
  "apiKey": "<your-api-key>"
}

This is right when OpenCode and Lemonade run on the same machine.

Remote example

"options": {
  "baseURL": "https://your-lemonade-host.example.com/api/v1",
  "apiKey": "<your-api-key>"
}

For OpenCode, unlike Claude, the OpenAI-compatible base path should include /api/v1.

Sample remote OpenCode config

Based on your existing file, with the provider pointed at a remote Lemonade host.

{
  "$schema": "https://opencode.ai/config.json",
  "provider": {
    "Lemonade": {
      "npm": "@ai-sdk/openai-compatible",
      "name": "Lemonade",
      "options": {
        "baseURL": "https://your-lemonade-host.example.com/api/v1",
        "apiKey": "<your-api-key>"
      },
      "models": {
        "Qwen3-Coder-Next-GGUF": {
          "name": "Qwen3-Coder-Next-GGUF",
          "maxInputTokens": 250000
        },
        "Qwen3-Coder-30B-A3B-Instruct-GGUF": {
          "name": "Qwen3-Coder-30B",
          "maxInputTokens": 250000
        }
      }
    }
  }
}
Verify First

Check the remote OpenAI-compatible routes before editing config.

OpenCode is easier to debug than Claude because its config is explicit. If these calls succeed, the client usually follows immediately.

1. Health

curl https://your-lemonade-host.example.com/api/v1/health \
  -H "Authorization: Bearer <your-api-key>"

Confirm Lemonade is up and recent enough for the features you expect.

2. Models

curl https://your-lemonade-host.example.com/api/v1/models \
  -H "Authorization: Bearer <your-api-key>"

Use the exact model IDs returned here inside the OpenCode provider block.

3. Chat endpoint

curl https://your-lemonade-host.example.com/api/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <your-api-key>" \
  --data '{"model":"Qwen3-Coder-Next-GGUF","messages":[{"role":"user","content":"hi"}],"max_tokens":8}'

If this returns a completion payload, OpenCode has what it needs.

Common Mistakes

OpenCode and Claude differ in one important way.

Claude

Origin-only base URL

Claude appends /v1/messages, so its ANTHROPIC_BASE_URL should be the server origin only.

vs
OpenCode

OpenAI-compatible base path

OpenCode uses the OpenAI provider config, so its baseURL should include /api/v1.

Troubleshooting

Most failures reduce to path, auth, or model naming.

Wrong base path

If you use the server origin without /api/v1, OpenCode will target the wrong route family.

Wrong auth header expectation

If your Lemonade host is configured for bearer auth, keep using the token as the OpenAI-compatible API key value in OpenCode.

Friendly name vs model ID

The name shown inside your OpenCode config can be friendly, but the top-level model key still needs to line up with the remote model ID.

Cloudflare Note

For remote OpenCode, prefer VPN over a public Cloudflare-protected path.

Why VPN is the safer default

OpenCode is a machine client making direct API requests, not a browser session clicking through challenge pages. If you put Lemonade behind public Cloudflare protection, API traffic can be challenged or shaped differently from normal website traffic. A VPN avoids that entire class of problems and keeps the API private.

Recommended setup

If you want to connect remotely, use VPN and point OpenCode at the private Lemonade address over that tunnel. That keeps authentication simple, avoids HTML interstitials in front of /api/v1/*, and is usually more reliable for streaming and repeated coding-agent requests.

Typical symptom

The website works in a browser, but OpenCode fails with unauthorized, timeout, HTML response, or generic provider errors because something at the edge returned browser-oriented traffic instead of JSON.

Good deployment pattern

Keep the public website public, but access the Lemonade API over VPN or another private network path whenever possible. That is a better fit for OpenCode than exposing the coding endpoint directly to the internet.

Fast sanity check

If curl to /api/v1/models or /api/v1/chat/completions returns HTML instead of JSON, the problem is usually your network path or proxy edge, not OpenCode or Lemonade.

Bottom Line

For OpenCode, remote setup is just a clean OpenAI-compatible config.

Use the remote /api/v1 base URL, carry over the right token, match the model IDs, and verify with one chat completion request before opening the app.
