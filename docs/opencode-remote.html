Remote OpenCode

Point OpenCode at a remote Lemonade server.

OpenCode uses an OpenAI-compatible provider config. For remote use, that means supplying the correct baseURL (including /api/v1), a valid API key, and model IDs that exactly match what the remote server reports.

remote-provider-overview

A

Provider config drives everything

The provider block in opencode.json sets the remote URL, auth token, and which model IDs appear in the session picker.

B

Lemonade exposes an OpenAI-compatible API

Use the @ai-sdk/openai-compatible npm provider. Your baseURL must include the /api/v1 path segment — not just the server origin.

C

Model keys must match the server exactly

The top-level key inside models is what OpenCode sends as the model field. It must match the ID returned by /api/v1/models on the remote server.

Config

Replace localhost with the remote base URL.

The provider block structure stays the same between local and remote. The only required change is swapping http://localhost:8000 for the public or private hostname where Lemonade is running.

Local example

"options": {
  "baseURL": "http://localhost:8000/api/v1",
  "apiKey": "<your-api-key>"
}

Works only when OpenCode and Lemonade run on the same machine. Do not use this when Lemonade is on a different host.

Remote example

"options": {
  "baseURL": "https://your-lemonade-host.example.com/api/v1",
  "apiKey": "<your-api-key>"
}

OpenCode sends requests to {baseURL}/chat/completions, so /api/v1 must be part of the base URL — unlike the Claude integration, which uses a bare origin.

Sample remote OpenCode config

Complete working config with the provider pointed at a remote Lemonade host. Replace the hostname, API key, and model IDs with values from your deployment.

{
  "$schema": "https://opencode.ai/config.json",
  "provider": {
    "Lemonade": {
      "npm": "@ai-sdk/openai-compatible",
      "name": "Lemonade",
      "options": {
        "baseURL": "https://your-lemonade-host.example.com/api/v1",
        "apiKey": "<your-api-key>"
      },
      "models": {
        "Qwen3-Coder-Next-GGUF": {
          "name": "Qwen3-Coder-Next-GGUF",
          "maxInputTokens": 250000
        },
        "Qwen3-Coder-30B-A3B-Instruct-GGUF": {
          "name": "Qwen3-Coder-30B",
          "maxInputTokens": 250000
        }
      }
    }
  }
}
Verify First

Verify the remote endpoints before editing your config.

Run these three curl commands from the machine running OpenCode. If all three return valid JSON, the provider config will work without further changes.

1. Health

curl https://your-lemonade-host.example.com/api/v1/health \
  -H "Authorization: Bearer <your-api-key>"

Should return {"status":"ok"}. A non-200 response or connection error means the server is unreachable from this network.

2. Models

curl https://your-lemonade-host.example.com/api/v1/models \
  -H "Authorization: Bearer <your-api-key>"

Returns a JSON list of available models. Copy the id field values exactly — these are the strings you must use as top-level keys in the OpenCode models block.

3. Chat endpoint

curl https://your-lemonade-host.example.com/api/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <your-api-key>" \
  --data '{"model":"Qwen3-Coder-Next-GGUF","messages":[{"role":"user","content":"hi"}],"max_tokens":8}'

Should return a streaming or non-streaming completion JSON. If this succeeds, OpenCode's inference requests will succeed too. An HTML response here means a proxy or firewall is intercepting the request.

Common Mistakes

OpenCode and Claude use different base URL conventions.

Claude

Origin-only base URL

The Claude SDK appends /v1/messages internally, so ANTHROPIC_BASE_URL must be the bare server origin (e.g. https://host.example.com) with no path suffix.

vs
OpenCode

Full base path including /api/v1

The @ai-sdk/openai-compatible provider appends only the endpoint name (e.g. /chat/completions), so baseURL must include /api/v1 — for example, https://host.example.com/api/v1.

Troubleshooting

Most failures come down to path, auth, or model ID mismatch.

Missing /api/v1 in base URL

If baseURL is the bare origin, OpenCode sends requests to /chat/completions instead of /api/v1/chat/completions. Lemonade returns 404, and OpenCode shows a generic provider error.

Incorrect API key

When LEMONADE_API_KEY is set on the server, every request must include a matching Authorization: Bearer <key> header. OpenCode sends the apiKey field as that bearer token automatically. A wrong or missing key produces a 401 response.

Display name vs model ID

The name field inside each model entry is a display label for the OpenCode UI. The top-level key for that entry is what gets sent in the model field of every API request. If the key does not match an ID from /api/v1/models, Lemonade returns a model-not-found error.

Cloudflare Note

Use VPN instead of routing API traffic through a public Cloudflare-protected path.

Why Cloudflare can break API traffic

OpenCode is an automated API client, not a browser. Cloudflare's bot-protection features — including JS challenges and Bot Fight Mode — can intercept API requests and return an HTML challenge page instead of JSON. The OpenAI SDK then fails to parse the response, and OpenCode displays a generic provider error. A VPN bypasses Cloudflare entirely and keeps the API on a private network path.

Recommended setup

Connect via VPN and point OpenCode at the private Lemonade address over that tunnel. This avoids HTML interstitials in front of /api/v1/*, keeps authentication straightforward, and is more reliable for the long-lived streaming connections that coding agents generate.

Typical symptom

The Lemonade web UI loads fine in a browser, but OpenCode fails immediately with an "invalid response" or "unexpected token" error. This happens because the edge node returned an HTML challenge page rather than a JSON API response.

Recommended deployment pattern

Keep any public-facing Lemonade web UI behind Cloudflare if you want, but expose the API port only over VPN or a private network. Avoid routing /api/v1/* through a Cloudflare proxy with bot protection enabled.

Quick sanity check

Run curl https://your-lemonade-host.example.com/api/v1/models from the same machine as OpenCode. If the response starts with <!DOCTYPE or <html instead of {"object":"list", the problem is your network path or edge proxy — not OpenCode or Lemonade itself.

Bottom Line

Remote OpenCode is a clean OpenAI-compatible config with the right base URL.

Set baseURL to your remote host with the /api/v1 path, set apiKey to your token, match model keys to the IDs from /api/v1/models, and confirm the chat completions endpoint responds before opening OpenCode.
