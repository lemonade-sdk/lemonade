## Agent 1 — Explore (recipe documentation + OmniRouter deep-dive)

**Routed because:** Before any terminology rename in the prototype, we needed a clear picture of how "recipe" is already used in the codebase and what OmniRouter actually is. Pure read-only research — no domain judgment yet.

**Mode:** sync (downstream synthesis was blocked on findings)

**Files authorized to read:**
- `docs/api/lemonade.md`
- `docs/guide/cli.md`
- `docs/dev/omni-router.md` (and adjacent `docs/dev/*`)
- `src/cpp/include/lemon/recipe_options.h`
- `src/cpp/server/recipe_options.cpp`
- `src/cpp/cli/` (for `lemonade import` recipe fetch)
- `src/app/src/` Tauri/React renderer (for `collectionModels.ts` wiring)

**Files produced:** None (research only — findings returned to Coordinator).

**Outcome (findings handed to Coordinator):**

1. **Engine-level recipe schema IS documented** in `docs/api/lemonade.md` + `docs/guide/cli.md` + `recipe_options.cpp`. The codebase concept of "recipe" = engine id (`llamacpp` / `flm` / `ryzenai-llm` / …) plus per-engine options applied at `/load` time.
2. **User-facing preset concept is NOT documented anywhere.** No prior precedent for a portable, model-less, sampling-bearing bundle in any docs surface.
3. **`lemonade import` CLI already supports remote recipe fetch** from the `lemonade-sdk/recipes` GitHub repo. Relevant precedent if/when we wire a "Browse community presets" feature in the UI.
4. **OmniRouter** is Lemonade's multimodal agentic pattern. Exposes modalities as OpenAI-compatible tools keyed by capability labels. Collections (Ultra / Lite) are 4-model bundles with `recipe: "collection"`. Already wired in the Tauri app via `collectionModels.ts` and consumed in `ChatWindow.tsx` — but with near-zero user-facing onboarding ("this is what OmniRouter does" is invisible to the user).
5. **Three-way "recipe" collision identified:**
   - Engine id (codebase, stable)
   - Collection meta-recipe (`recipe: "collection"`, OmniRouter)
   - User-facing preset bundle (the v1.2 prototype's "Recipe")

## Agent 2 — Coordinator synthesis → Kyle decision

**Not a spawn — inline coordinator work.** Folded Explore's findings into four considerations and presented them to Kyle:

1. **Terminology lane** — rename to "Preset" to free "Recipe" for the two existing codebase meanings.
2. **Capability-vs-engine keying** — switch the compatibility model from engine-list to capability-label-list to align with OmniRouter's tool-based routing.
3. **OmniRouter UI gap** — Collections load, but the user has no surface that says "this is OmniRouter, here's what it does."
4. **`lemonade-sdk/recipes` import target** — a Browse-community-presets feature has obvious infrastructure already present.

**Recommendation:** Rename to Presets + capability-keyed.

**Kyle's verdict (verbatim):** *"Presets is fine. capability keyed please."*

**Explicitly deferred** (narrow approval): Browse community feature, OmniRouter UI onboarding. These are separate workstreams.
