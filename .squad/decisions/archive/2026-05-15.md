### 2026-05-15T18:00:00Z: UI POC framework recommendation — Svelte 4 + webpack on one panel
**By:** Mattingly (UI / Frontend)
**Status:** Recommendation, pending Kyle's decision on open questions

**What:**
- Recommend POC the new UI in **Svelte 4 + webpack + svelte-loader**, NOT SvelteKit, NOT Flutter, NOT a full React rewrite.
- POC scope: ONE panel (suggested: `TranscriptionPanel` or `EmbeddingPanel`) re-implemented in Svelte and served at `/app-next` (web) plus a debug-menu route inside the existing Tauri window (desktop). Existing `/app` and existing Tauri renderer remain untouched.
- New parallel build directories: `src/app-next/` (Svelte source) and `src/web-app-next/` (Svelte+webpack build config), plus `BuildWebAppNext.cmake` cloned from `src/web-app/BuildWebApp.cmake`.
- Reuse `tauriShim.ts` as a framework-agnostic TS module; reuse the C++ server's `window.api` mock injection unchanged.

**Why:**
- **Flutter is not viable.** Flutter Desktop ships its own Skia/Impeller renderer — it does not embed in WebView2/WKWebView/webkit2gtk, so adopting it means replacing Tauri entirely. Flutter Web bundles are heavy (~2–4 MB gzipped) hurting tray-launch UX. Debian packaging would require vendoring the Flutter engine, contradicting invariant #12.
- **SvelteKit is not viable under Debian rules.** `@sveltejs/kit` and vite are not reliably present in `/usr/share/nodejs`. Adopting them forces either pre-built bundle vendoring or violating `USE_SYSTEM_NODEJS_MODULES`. Either path breaks invariant #12.
- **Svelte 4 (compiler) and svelte-loader (webpack integration) ARE in Debian trixie.** This preserves the existing build pipeline: webpack stays, `BuildWebApp.cmake` staging stays, `USE_SYSTEM_NODEJS_MODULES` semantics stay. Only the rule chain and resolve extensions change.
- **Svelte's reactivity model fits the app shape** (CRUD over HTTP, lots of small independent state, streaming responses) better than the current React-with-context approach. Kyle has Svelte experience.
- **React isn't broken.** The pain in `ChatWindow.tsx` / `ModelManager.tsx` is component size + state sprawl, not framework choice. The POC's job is to determine whether Svelte's leverage is worth the migration cost.

**Hard constraints honored:**
1. ✅ Single codebase, both surfaces — Svelte bundle runs identically in browser and Tauri webview.
2. ✅ Debian `/usr/share/nodejs` filter — `node-svelte` + `node-svelte-loader` + existing webpack stack.
3. ✅ Many-clients-one-server — `tauriShim.ts` boundary preserved.
4. ✅ Desktop is on-demand — no Tauri lifecycle changes.
5. ✅ Side-by-side coexistence — existing `src/app/` and `src/web-app/` untouched.

**Gate criteria for POC → migration decision:**
1. `svelte-loader` + system `node-svelte` builds cleanly under `USE_SYSTEM_NODEJS_MODULES=1` in a Debian trixie container.
2. Resulting bundle is measurably smaller than the React equivalent.
3. Velocity check: did Kyle finish the panel faster than the React rewrite would have taken?
4. `salsa-ci` still passes with the second pipeline added.

**Explicit non-decisions (waiting on Kyle):**
- Svelte 4 (Debian-friendly) vs Svelte 5 / runes (requires bundle vendoring) — see open question 1.
- Which specific panel to port first — see open question 2.
- Coexistence horizon between React and Svelte renderers — see open question 4.
- Whether to mirror existing hooks 1:1 as Svelte stores or redesign them during port — see open question 5.

**Fallback plan:** If the POC reveals a Debian-pipeline blocker that doesn't have a clean fix, fall back to "stay React, refactor the giant components." That fallback is independently valuable.

### 2026-05-15T18:00:00Z: UI framework packaging verdict
**By:** Kranz (Build & Release)
**What:**
- **React (status quo):** No packaging change. Already validated on every target.
- **Svelte (vanilla):** Tolerable. `node-svelte` and `node-svelte-loader` are NOT in Debian's `/usr/share/nodejs`. Must add the `webpack` toolchain inputs to `debian/control` as vendored or pre-built artifacts, OR shift to the "pre-build and ship" pattern for the .deb. Manageable.
- **SvelteKit:** Rejected for Debian packaging. Dep tree (vite, rollup, adapter-static, kit internals) has zero presence in Debian's nodejs packages. Would require committing or downloading a pre-built `dist/` for the .deb — a significant departure from current `debian/rules`.
- **Flutter Web:** Acceptable IF we adopt the pre-built-artifact pattern. Build runs through Dart SDK (not Debian-packaged for builds) → cannot satisfy `dh_auto_configure` with system tools. Must ship pre-built static assets.
- **Flutter Desktop:** Rejected. Own renderer (Skia/Impeller). Does NOT embed inside Tauri's WebView2/WKWebView/webkit2gtk. Replacing Tauri = replacing the desktop binary + bundler + installer integration. Out of scope for a POC.

**Why:** Debian native packaging requires either `/usr/share/nodejs` modules OR a clean pre-built artifact path. Today's `debian/rules` invokes `cmake+ninja` which runs webpack at .deb build time against system nodejs modules — so any framework whose toolchain isn't in `/usr/share/nodejs` forces a packaging-architecture shift (pre-build outside Debian context, ship the bundle as data). That shift is feasible but is a real engineering cost the POC must own.

**Recommendation for POC scope:** Keep the side-by-side POC focused on Svelte (vanilla) and/or Flutter Web. Defer SvelteKit unless we explicitly accept the pre-built-artifact packaging change. Hard-reject Flutter Desktop — it breaks the Tauri embedding contract that the desktop binary depends on.

### 2026-05-15T18:00:00Z: UI ↔ `lemond` contract (POC source of truth)
**By:** Liebergot (C++ Server Core)
**Requested by:** Kyle Poineal
**What:** Canonical contract document defining what the new UI must talk to. `lemond` is frozen for the UI POC — this is the immutable surface Mattingly's POC must conform to. Any deviation requires a new decision.

#### 0. Topology (read this first)

- `lemond` is a pure HTTP server. It runs as its own process (Windows: embedded in `LemonadeServer.exe`; macOS/Linux: standalone, optionally driven by `lemonade-tray`).
- The UI — Tauri desktop, browser web app, or anything new — is a **client**. It speaks HTTP to `lemond`. It does NOT embed `lemond`.
- A single `lemond` can serve many clients (invariant #11). Per-client state (zoom, layout, theme, base URL, API key) lives in the client, NOT in `lemond`. There is no server-side `app_settings.json` endpoint and there must never be one.
- Discovery: local clients can find a running `lemond` via UDP beacon (the CLI does this). Remote clients use an explicit base URL.

#### 1. REST endpoint inventory

**Quad-prefix invariant (invariant #1)** — every core endpoint registered FOUR times:
- `/api/v0/{endpoint}` (legacy)
- `/api/v1/{endpoint}` ← **recommended for new UI**
- `/v0/{endpoint}` (legacy short)
- `/v1/{endpoint}` ← OpenAI SDK / LiteLLM compatibility

Ollama API exception — routes live under `/api/` with no version prefix. Anthropic endpoint is at `/v1/messages` (single prefix).

**1a. Model management** — `GET models`, `GET models/{id}`, `POST pull`, `GET pull/variants`, `POST load`, `POST unload`, `POST delete`, `POST params`, `POST install`, `POST uninstall`.

**1b. Inference** — `POST chat/completions` (SSE), `POST completions`, `POST embeddings`, `POST reranking`, `POST responses`, `POST audio/transcriptions`, `POST audio/speech`, `POST images/generations`, `POST images/edits`, `POST images/variations`, `POST images/upscale`, `GET slots`, `POST slots/{id}`.

**1c. Server status & system** — `GET health` (returns `{status, version, model_loaded, all_models_loaded, max_models, websocket_port}`), `GET stats`, `GET system-info`, `GET system-stats`, `POST log-level`, `GET /live` (single registration, NOT quad-prefixed).

The renderer already fetches `/health`, `/system-stats`, `/system-info` directly via fetch — the old Tauri `system_info.rs` proxy was removed as redundant. The new UI should do the same.

**1d. Ollama-compatible API** — single prefix `/api/`, no versioning. Endpoints: `POST /api/chat`, `POST /api/generate`, `GET /api/tags`, `POST /api/show`, `DELETE /api/delete`, `POST /api/pull`, `POST /api/embed`, `POST /api/embeddings`, `GET /api/ps`, `GET /api/version`. Stubs (501): `/api/create`, `/api/copy`, `/api/push`, `/api/blobs/{digest}`.

**1e. Anthropic-compatible API** — `POST /v1/messages` (Anthropic message completion, tool use, SSE streaming).

**1f. Internal endpoints (NOT for the UI under normal use)** — `POST /internal/shutdown`, `POST /internal/set`, `GET /internal/config`, `POST /internal/cleanup-cache`. Admin-only (`LEMONADE_ADMIN_API_KEY` when auth is enabled).

#### 2. WebSocket Realtime API

- **Discovery:** `GET /health` returns `websocket_port`. WS server binds to an OS-assigned port starting at 9000+. **UI must read `websocket_port` from `/health` — do not hardcode.**
- **Host:** same host as the HTTP server.
- **Endpoints multiplexed on the same port:**
  - `/realtime` — OpenAI Realtime API protocol for streaming audio transcription
  - `/logs/stream` — server-side log stream (subscribe with optional `after_seq` for replay)
- **Protocol:** standard WebSocket. Messages are JSON. libwebsockets may fragment frames; server reassembles internally.
- **Auth:** same as HTTP. Pass via `Authorization: Bearer {key}` header or URL arg.

#### 3. The `window.api` shim contract

Renderer was originally written against Electron's `contextBridge`. The Tauri shim (`src/app/src/renderer/tauriShim.ts`) preserves that contract by installing a `window.api` object on the renderer. The web build replaces it with a server-injected mock. **The new UI's adapter layer must implement these methods twice — once for Tauri, once for the browser.**

**3a. Identity / platform** — `isWebApp` (`false` Tauri / `true` web mock), `platform` (Tauri: `invoke('get_platform')` cached; web: `navigator.platform || 'web'`).

**3b. Window controls (Tauri-only; web build = no-ops)** — `minimizeWindow()`, `maximizeWindow()`, `closeWindow()`, `onMaximizeChange(cb)`, `updateMinWidth(w)`, `startResizeDragging(dir)` (webkit2gtk needs renderer-painted resize regions), `zoomIn()` / `zoomOut()` (web mock mutates `document.body.style.zoom`).

**3c. Native bridges (Tauri-only; web has fallbacks)** — `writeClipboard(text)` (Tauri plugin; web fallback `navigator.clipboard.writeText`), `openExternal(url)` (Tauri plugin; web `window.open(url, '_blank')`).

**3d. Per-client settings (Tauri = native storage; web = localStorage)** — `getSettings()` (Tauri: `invoke('get_app_settings')` reads `app_settings.json`; web: reads `localStorage['lemonade-settings']`), `saveSettings(s)`, `onSettingsUpdated(cb)`. **Load-bearing invariant #11 check.** Settings are per-client. Never proxy these through HTTP to `lemond`.

**3e. Server discovery & connection (Tauri = UDP beacon + native config; web = window.location)** — `discoverServerPort()` (Tauri: UDP beacon scan), `getServerPort()` (web: `parseInt(window.location.port) || 13305`), `getServerBaseUrl()` (Tauri, can be remote), `getServerAPIKey()`, `onServerPortUpdated(cb)`, `onConnectionSettingsUpdated(cb)` (`{base_url, api_key}`).

**3f. Misc** — `getLocalMarketplaceUrl()`, `signalReady()` (Tauri: `invoke('renderer_ready')`), `onNavigate(cb)` (deep-link `{view, model}`), `restartApp()` (web-only convenience, `window.location.reload()`).

**3g. Web-mock injection point** — `src/cpp/server/server.cpp` (around line 611) reads `resources/web-app/index.html`, inserts a `<script>window.api = {...}</script>` block just before `</head>`, and sets no-cache headers. The mock is intentionally minimal.

#### 4. Auth & CORS

- **`LEMONADE_API_KEY` env var** → all regular API routes (quad prefixes + Ollama + Anthropic) require `Authorization: Bearer {key}`. Pre-routing handler rejects with 401 if unset/wrong.
- **`LEMONADE_ADMIN_API_KEY` env var** → full access including `/internal/*`. Clients prefer admin key if both set (Tauri host does this).
- **No keys set** → no auth required (default dev mode).
- **CORS** — `Access-Control-Allow-Origin: *`, methods `GET, POST, PUT, DELETE, OPTIONS`, headers `Content-Type, Authorization`. Enabled on all routes. The browser web app build relies on this; do not break it.
- **WebSocket** — same key, same precedence. Pass via `Authorization` header or URL arg.

#### 5. Things the new UI MUST NOT do

1. **Move per-client state into `lemond`.** Zoom, layout, theme, panel widths, base URL, API key — all client-side. Invariant #11.
2. **Embed `lemond` in the desktop binary.** Desktop app is on-demand. `lemond` runs independently. UI discovers a running server; it does not spawn one. Invariant #13.
3. **Drop, rename, or single-prefix any quad-registered endpoint.** Invariant #1.
4. **Move `app_settings.json` behind an HTTP endpoint.** Settings stay per-client.
5. **Hardcode the WebSocket port.** Always read it from `/health`.
6. **Assume `lemond` and the UI are co-located.** Web build runs in a browser pointed at a possibly-remote server. Tauri build can be configured with a remote base URL. Use the discovered/configured base URL for every fetch.

#### 6. Things the new UI gets for free

1. **Server discovery** — Tauri: `invoke('discover_server_port')` UDP beacon scan; web: same origin.
2. **The injected `window.api` mock** — keeping the shim shape gives web/desktop parity for free.
3. **Quad-prefix flexibility** — pick `/api/v1/` everywhere.
4. **OpenAI SDK compatibility** — `/v1/` prefix means OpenAI JS SDK and LiteLLM Just Work. Same for Anthropic at `/v1/messages` and Ollama clients at `/api/*`.
5. **SSE streaming** — chat completions, completions, Anthropic messages all support `stream: true`.
6. **WebSocket multiplexing** — one WS port serves both Realtime audio and log streaming.

#### 7. Summary for Mattingly

Build against `/api/v1/` (default), `GET /health` for discovery (incl. `websocket_port`), and the `window.api` contract in §3. Tauri adapter wires §3 to `invoke()`/`listen()`. Web adapter accepts the injected mock from `lemond` (or replaces it with a richer browser implementation respecting the same shape). Settings stay client-local. Auth is a single `Authorization: Bearer` header. Streaming is SSE for text, WebSocket for audio/logs.

**Why:** Single document Mattingly can cite when choosing a framework and designing the adapter layer. Codifies the existing contract so the POC doesn't drift into backend changes that would violate the project working rules.

The packaging arch change is then ALSO a cost of vanilla Svelte. My prior framework recommendation rested on a Debian-availability claim the build authority contradicted, and I missed it.

## Per-option Lead verdict (as requested)

## Strategic case (first principles)

1. **The project does not treat the UI as a strategic asset.**
   `docs/dev/philosophy.md` is explicit: *"Lemonade has a GUI for two reasons only: to
   show you what's possible, and to help you manage models and backends across
   connected apps."* `AGENTS.md` is explicit: *"UI/frontend changes are handled by core
   maintainers only."* The mission ("make local AI accessible") flows through `lemond`
   and the OpenAI-compatible API surface. The GUI is a means. Investing engineering
   capital in UI-framework velocity is investing in the wrong layer. The fungibility
   tenet applies to **backends**, not to renderers.

2. **Invariant #12 is not a tolerance question — it's a maintenance question.**
   Kranz's report says Svelte 4 is *tolerable* under Debian (`node-svelte`,
   `node-svelte-loader` either exist or can be added to `debian/control`). "Tolerable"
   ≠ "free." Every entry in `contrib/debian/control` is something Mario Limonciello (or
   his successor) keeps in sync with upstream forever. The current `control` is a
   tightly-curated React+webpack set that has been validated end-to-end. Doubling that
   surface to also cover Svelte is a permanent maintenance tax on someone who is not on
   this squad. The cost lives downstream of the people deciding to pay it.

3. **The success criterion for the Svelte POC cannot answer the question being asked.**
   *"One panel works in Svelte"* tells us Svelte compiles. It does not predict
   maintainability at 50 panels, contributor onboarding cost, debug surface across
   Tauri+browser, or whether the next four maintainers will tolerate two frameworks.
   N=1 panel results — especially a panel the framework's strongest advocate built — are
   the classic POC trap. We would be paying real engineering cost for a verdict we
   already know will be ambiguous.

4. **The actual product pain is observable and falsifiable.**
   `ModelManager.tsx` is 75,054 bytes. `ChatWindow.tsx` is 13,858 bytes. Whatever
   friction Kyle is feeling, the magnitude is concentrated in a single file. A React
   decomposition POC has a concrete, measurable success criterion: *"ModelManager.tsx
   becomes N sub-components, each under 10 KB, with state hoisted to clear owners, and
   the next feature lands faster."* That answers the velocity question directly. If
   that refactor lands and Kyle still feels the framework is in the way, we have
   evidence — not a hunch — to justify a Svelte exploration.

5. **Reversibility cuts both ways.**
   Mattingly's correct that the POC is cheap to abandon. But "cheap to abandon" is also
   what tempts a team to *not* abandon it. Once `src/app-next/` exists, has a build
   target, has a panel, and has a CI lane, the marginal cost of porting "just one
   more" panel feels low at every step — until two years later the tree has Svelte and
   React and no one is allowed to retire either.

## Risks I'm accepting by rejecting (B/C)

- **We never validate Kyle's intuition that Svelte would be cleaner.** Real risk. The
  React refactor POC mitigates this by giving Kyle's intuition a concrete head-to-head:
  if decomposing `ModelManager` is itself painful in a way that Svelte's reactivity
  model would have prevented, we learn that the right way (with evidence).
- **Kyle's Svelte experience goes unused on this project.** True. That's a personal
  cost, not a project cost. The Lead's job is to optimize the project, not the
  individual maintainer's preferred stack.
- **The "Tauri + Svelte" ecosystem combo argument is left on the table.** Weak
  argument. Tauri is framework-agnostic at the webview boundary; community templates
  exist for every framework. We gain nothing by adopting a popular pairing if the
  pairing isn't load-bearing for *our* problem.

## Risks I would be accepting if I sanctioned (A)

- **Permanent doubling of the UI dependency surface in `contrib/debian/control`.** Paid
  by Debian maintainer, not us. That's worse, not better — externalized cost is the
  hardest kind to take back.
- **Permanent contributor-onboarding tax.** Every new UI contributor needs to learn
  which framework owns which panel and why.
- **Mattingly + Kranz + Kyle + me spend 2–3 weeks on a POC whose success metric is
  weaker than what a React refactor POC would deliver in the same time.** Opportunity
  cost is real product features or test coverage Haise could be building.
- **Bus-factor risk.** Mario maintains `lemonade` in Debian; he is not on this squad
  and did not sign up for a second renderer toolchain.

## Decision rule — what would change my mind

I would re-open the Svelte question **if and only if** one or more of these holds:

1. The React decomposition POC lands, `ModelManager.tsx` is split cleanly, and the
   next feature added on top of it is still measurably slow / error-prone because of
   React-context propagation or rerender behavior — *with traces or profiles, not
   vibes*.
2. Mario (or another Debian maintainer) explicitly volunteers to own the Svelte
   toolchain entries in `control` long-term. That removes the externalized-cost
   objection.
3. A real product requirement appears that React + the current dep set cannot meet
   without an upgrade Debian doesn't ship (e.g., we need a feature that mandates a
   React 19+ library Debian doesn't have, and Svelte already covers it). Framework
   shopping for need, not for taste.
4. Kyle commits to personally maintaining the Svelte half forever, in writing, in
   `decisions.md`. (Honest framing: this is the "this is my project and I want this"
   override clause. It is a valid override. The Lead's job is to surface it as such,
   not to pretend it doesn't exist.)

## What the POC should be instead

Sanction Haise/Mattingly to spend the same budget on:

1. Decomposing `ModelManager.tsx` into per-concern sub-components (`ModelList`,
   `ModelRow`, `ModelActions`, `RecipePicker`, `FamilyGrouping`, `DownloadProgress`,
   `BackendStatus`, etc.). Each under 10 KB. State hoisted to clearly named owners.
2. Auditing the giant `useMemo` / `useEffect` chains in `ChatWindow.tsx` for any that
   should be hooks (`useModelType`, `useActivePanel`).
3. Adding renderer-level tests around the refactored boundary so future component
   splits don't regress. (Coordinate with Haise.)
4. Writing the result up in `docs/dev/web-ui.md` as the canonical React component
   guideline so the next 75 KB component is caught at review.

Same budget. Concrete, measurable outcome. Zero `debian/control` churn. Zero new
framework on contributors' plates. Directly addresses the actual pain.

## Acknowledging the second opinion

Mattingly is right that *if* we were going to do a non-React POC, Svelte 4 + webpack is
the only Debian-compatible choice and her topology design is sound. Her recommendation
is the correct answer to the question *"which framework, if any?"* My verdict is to
the prior question *"should we be picking a framework at all?"* — and the answer is no,
not yet, not on this evidence.

**Why:** Strategic ROI does not justify the engineering cost of a Svelte POC given (a)
the project's stated philosophy that the UI is a means, not an asset; (b) the
externalized maintenance cost on Debian packaging; (c) the bus-factor profile of the
maintainer pool; (d) the fact that the observable pain is component-design pain, not
framework pain, and is directly addressable in React; and (e) the Svelte POC's
success criterion cannot actually answer the strategic question being asked.

## Section 1 — Critique of the current UI

The shipping UI is **functional, fast, and consistent** — that's the floor. It's dark,
the panels load, the chat streams, the status bar tells the truth. As a v1 of a
locally-hosted multi-backend, multi-modal AI server's GUI, it works. The question
Kyle is asking is the ceiling, not the floor.

**Information architecture.** The icon-rail → sidebar → main pane → status-bar
structure is a VS Code skeuomorph. That was a deliberate call (per
`docs/dev/philosophy.md`: *"Most users are familiar with VS Code, making this
interface simultaneously minimal and discoverable."*) The decision was right at v9 but
has now hit its ceiling: VS Code's three columns serve *file → workspace → editor*,
which is a single user task. Lemonade's three columns serve *unrelated* tasks — Model
Manager, Backend Manager, Marketplace, Chat — that the user switches between, not
combines. So the rail forces the user to drop one task to start another. The chat
pane is empty whenever the user is doing model/backend/marketplace work, which is
≥50% of the active surface area sitting unused.

**Visual hierarchy.** The eye lands on the list (left/middle), not the chat. The
chat — which is the actual product — visually loses to the manager panes. The "LLM
CHAT" cap-locked label and a refresh icon are the only chat affordances above the
fold, against a sidebar pulsing with green dots, version numbers, badges. The
attention is in the wrong place.

**Density.** Inconsistent. The status bar is dense (7 metrics, one row, micro-type).
The chat empty state is the opposite — a single lemon emoji and two words on a
1440-px-wide canvas. Model Manager sits in the middle: dense data, narrow column,
constant truncation. The app reads as three different products written by three
different design sensibilities glued together.

**Empty states.** The most-seen view of the most-used surface is a lemon and a
title. Nothing about what the loaded model can do, suggested prompts, recent
threads, a featured model card, capabilities. Zero work being done.

**Affordance & feedback.** The colored dots on models have no legend. "Backend
update required" is buried in row text, not a banner. Truncated names. Git SHAs
exposed to end users. The active-tab indicator on the icon rail is invisible in the
screenshot — I have to guess which view I'm on from the title in column 2.

**Visual style.** Dark Material-ish with some custom touches (the lemon, the cap-
locked section headers). It isn't *bad*. It isn't a *design system* either —
there's no clear type scale, the cap-lock headers feel like a stylistic tic rather
than a tier, and accent color is inconsistent (orange in some places, green dots,
purple dots, white outlines for "not installed"). One designer-day of token work
would unify 70% of it.

**Pain by view.**
- *Model Manager:* truncation, legend-less dots, conflation of "active right now" with
  "downloaded and available" with "registry-known but not installed," and `user.*` prefix
  leaking on-disk naming into the UI.
- *Backend Manager:* `b8940`, `master-509-ab6afe8` exposed to humans. The whole concept
  of "backends" is a power-user mental model surfaced as a first-class nav item.
- *Marketplace:* 2008-directory layout with Visit/Guide buttons. Cards have nothing
  Lemonade-specific in them — they're the same as a landing page on a partner site.

**Top 3 most impactful issues, ranked:**

1. **The chat empty state does no work.** The most-seen surface of the most-used
   feature wastes its real estate. Highest ROI to fix.
2. **Information architecture flattens four unrelated jobs into one rail.** Backend
   Manager and Marketplace probably don't deserve top-level nav. Chat does. Models
   probably does. Everything else is one level deeper.
3. **No design system, no legend, no banner pattern, no consistent name treatment.**
   Visual coherence is one designer-day plus a token file. Highest "per hour invested"
   improvement.

## Section 2 — Competitive review

I fetched each site below. LM Studio, Open WebUI, Jan, Msty, Cherry Studio, GPT4All
all came back; Msty redirected from `msty.app` → `msty.ai`. I'll cite URLs inline.

**LM Studio (https://lmstudio.ai)** — the dominant local-LLM GUI. Their hero
screenshot has *one* surface: a chat with a model card subtly placed top-right and a
model dropdown in the title bar. No icon rail. No persistent backend manager. They
recently shipped `llmster` (LM Studio without the GUI for headless deployment) and
"LM Link" (remote-connect to a running LM Studio). The IA decision is clear: chat is
the product; managers are progressive disclosure.
*Steal:* Hoist the model selector into the title bar; it's a primary control, not a
footer dropdown.
*They got wrong:* still developer-coded language ("local LLMs," "tokens per second"
front-of-house). Lemonade can be friendlier.

**Open WebUI (https://openwebui.com)** — community-platform play. 401K members,
shared prompts/tools/functions, models browseable from the app. This is the *most
underrated UX move in the category*: the model browser isn't a registry of files,
it's a feed with reviews, descriptions, capability tags.
*Steal:* The marketplace should be a model/persona/prompt feed, not a directory of
third-party apps. Lemonade's "Marketplace" semantics are weak — visit-an-external-app
isn't what users want.
*They got wrong:* it's a *web* app, so it leans into account/community/SSO. Lemonade is
private-first and local-first; we don't follow them into account-land. Keep the feed,
drop the account.

**Jan (jan.ai)** — explicitly the design-forward entrant. The wall of user
testimonials calls it out: *"simple and cute and pretty," "cleanest interfaces and
local implementations," "the most beautiful native chat app ever made."* That's the
"Apple-jealous" target tone. Jan's surface is a single chat with a left rail of
*conversations*, a right inspector that slides in for model/persona/MCP config, and a
Memory concept ("things Jan keeps in mind").
*Steal:* conversation-rail-on-the-left is the obvious chat-as-product layout. And
Memory ("things Lemonade knows about you") is a real differentiator — it's *just*
localStorage but presented as a feature.
*They got wrong:* leans too hard on charm — the "things Jan keeps in mind" card with a
Joe-the-Designer avatar verges on Clippy. Lemonade can be polished without being
cutesy.

**Msty / Msty Studio (msty.ai)** — "workspace built like a studio." Three products
now: Studio (chat + workflows + personas + knowledge stacks), Claw (autonomous agent),
Stack (centralized inference). Knowledge Stacks are RAG repos as first-class
citizens.
*Steal:* the "knowledge stack" framing — local docs attached to a conversation as a
nameable, reusable, swappable unit. Lemonade has the pieces (embeddings, reranker,
chat) and not the UX.
*They got wrong:* the product family is sprawling — Studio + Claw + Stack + Law +
calculators. Lemonade can avoid that by keeping one app, one mental model.

**Cherry Studio (cherry-ai.com)** — polished multi-provider client, 45K stars. Their
killer move: *side-by-side model comparison in a single chat*. Send one prompt, see N
models answer in parallel columns. Also 1000+ built-in assistants (personas), 300+
provider integrations.
*Steal:* side-by-side model comparison is *uniquely well-suited* to Lemonade — we are
the local server with multiple loaded models. Nobody else has the substrate to do
this cleanly. This is the differentiator.
*They got wrong:* the provider chooser is overwhelming — 50+ provider logos on the
homepage is choice paralysis. Lemonade has fewer providers (we ARE the provider) and
should celebrate that simplicity, not chase parity.

**GPT4All (nomic.ai/gpt4all)** — venerable, has LocalDocs (RAG). Site now reads as
enterprise-adjacent (the parent Nomic pivoted to AEC vertical). Visual design is
*deeply* dated — gray, dense, no hierarchy.
*Steal:* nothing visual. The LocalDocs *concept* matters (private documents grounding
chat), but Msty/Cherry/Open WebUI all do it better.
*They got wrong:* everything. This is what Lemonade must not become.

**The "Apple bar."** Apple Intelligence's surface — the writing tools popover, Image
Playground, Genmoji creator — is single-purpose, single-input, single-output. No
chrome. No mode switcher. No status bar. The chrome appears only when needed.
Activity Monitor / System Settings is the same: one navigation list, one detail pane,
and the detail pane *changes shape* per topic — graphs for CPU, list for processes,
form for users. Lemonade's panels are all the same shape (sidebar + list + detail),
which is exactly why Backend Manager and Marketplace feel like Model Manager in
trench-coats. The lesson: **let each surface have the shape its content demands.**

## Section 3 — Design principles for the redesign

Five principles, concrete enough to settle a tie in a design call:

1. **The chat is the product. The managers are tools.** Chat gets primacy of pixels,
   primacy of nav, primacy of keyboard. Everything else is progressive disclosure.
2. **Names, not git SHAs.** No `b8940`. No `master-509-ab6afe8`. Versions become "Latest"
   or "Update available" or a relative date. The SHA lives in a "Details" tooltip for
   people who care.
3. **Status is contextual, not omnipresent.** TPS / TTFT / RAM / CPU are valuable to
   power users and noise to first-timers. Default: a single connection dot + active
   model name. Power users opt in to the full strip.
4. **Loading is a state, not a screen.** No separate "Downloading…" page. Downloads
   appear inline on the model row with a progress ring and stay visible. Cancellable.
5. **One opinion at first launch, infinite tinkering after.** First launch picks a
   default backend and a default model for the user's hardware automatically. The
   Backend Manager exists for people who go looking.
6. **The model is a character, not a row.** The Model Manager should feel like a
   roster of *characters with capabilities*, not a list of files on disk.
7. **Capability is the user's noun, recipe is ours.** The user wants "vision" or "code"
   or "embeddings." `flm` / `ryzenai-llm` / `vulkan` are our naming, not theirs.
   Surface capability first, recipe second.

## Section 4 — Concrete proposals by surface

### Chat surface (the most-seen view)

**Empty state.** Replace the lonely lemon with a *hero card* that does work:

- Centered: large warm greeting ("Hey Kyle — what's on your mind?") with the user
  name from localStorage.
- Below the input: a row of **suggested capability chips** — *"Summarize a doc," "Code
  review," "Generate an image," "Transcribe audio."* Click → routes to the right
  model type. (We KNOW what's loaded; we can route.)
- Below that: **"Active models"** as 1–3 small cards, each with the model name, badge
  for capability (chat/vision/code/embed/TTS/image), TPS expectation, and a "Switch
  to" pill. This replaces the footer dropdown for empty-state.
- Far right (collapsed by default): **recent conversations** rail. Click to expand.

**Chat history.** Move it from "nonexistent" to a left rail that slides in/out. New
chat is a `⌘N` / `Ctrl+N` action with a dedicated button at the top of the rail. Each
conversation has a model-icon badge so users can see at a glance which thread used
which model. Conversations are localStorage (invariant #11).

**New chat affordance.** Always-visible "+ New chat" button at the top-left of the
conversation rail. Default model is whatever's currently loaded. `⌘N` from any
surface.

**Streaming response presentation.**
- Type scale up: 15px body, 14.5px UI elsewhere. Chat content gets the larger size.
- Code blocks: filename header bar with copy + collapse + language badge. Current
  highlight.js is fine; just frame it.
- Inline citations when the response used RAG (future Knowledge Stacks feature).
- Math: katex already in the stack. Looks fine.
- Attachments rendered above the input as chips with thumbnail, removable.

**Model selector.** Hoist to the title bar (LM Studio pattern). Format:
`{model name} ▾  ●` where the dot reflects load state. Click → drop-down with active
models on top, switchable in one click, plus "Manage models…" at the bottom.

**Multi-modality.** Per-message kind picker is wrong UX — users don't think "I want
to send an audio message"; they think "I want to talk to it." Use a single composer
with a `+` button that surfaces attachment types based on the *loaded model's
capabilities*. If a vision model is loaded, image attach shows up. If TTS is loaded,
a 🔊 toggle on response shows up. Otherwise hidden.

**Token / TPS / TTFT metrics.** Per-message inline as a faded line under the
response: *"44 tok/s · 3.62s to first token · 347 tokens."* Click to expand to full
detail. The status bar's job becomes a single pill: `● Connected — Qwen3-26B`.
Full status strip lives behind a chevron / `⌘.`.

### Model Manager

**One list, two zones, smart sort.** Top zone: **"Loaded"** (currently active, 1–N).
Below: **"Installed"** (downloaded but not loaded). Below that: **"Available"**
(registry-known, not installed). Same row template throughout. No expanding family
chevrons — family is *just* a filter chip.

**Row template:**
```
[icon] {Display name}                 {capability badges}  {device badge}  {size}  [Load ▸]
       {family · params · quantization}                                            [⋯]
```

Two lines, generous whitespace. No truncation — if the name is long, it wraps and
the row grows.

**Colored-dot problem.** Replace dots with **named capability badges**: `Chat`,
`Vision`, `Code`, `Embed`, `Rerank`, `Audio`, `TTS`, `Image`. Color is decorative,
not load-bearing. A small legend lives at the top of the panel collapsed by default.

**Cards vs rows.** Rows for the list. Cards only on *model detail* (click a row →
slide-over panel with description, capability matrix, recommended use cases,
sample prompts, license, link to HF). The slide-over is the Apple System Settings
pattern: list on left, detail on right, list stays visible.

**Search + filter chips at top.** Filter by capability (Chat/Vision/Code/Embed),
device (CPU/GPU/NPU), family (Qwen/Llama/Gemma…), size (<5 GB / 5–20 / 20+). The
chips replace the chevron-expand and the implicit `user.*` namespace.

**`user.*` prefix.** Hide it. The UI shows the display name; the on-disk namespace
is a `Details ▸` row in the slide-over.

**Install / load / delete flows.** All inline on the row. Progress is a thin
horizontal bar across the row's bottom border while downloading. Cancel is the
`⋯` menu. Delete confirms via a sheet, not a modal that owns the screen.

**Size labels.** Right-align them; right-align the device badge. Two-column right-
rail = scannable.

### Backend Manager

**Demote it.** Backend Manager probably shouldn't be a top-nav icon. Move it into
**Settings → Backends** as a sub-page. First-launch experience picks the best
backend for the user's hardware (we already detect GPU/NPU). Most users never need
to open it.

**If we keep top-level (Kyle's call):** redesign as a **device-first matrix** instead
of a backend-first list. Rows = devices (CPU / GPU / NPU). Columns = capabilities
(LLM / Audio / Image / TTS). Each cell = the installed backend variant for that
device × capability, with a clickable swap action. Version display is "Latest" or
"Update — May 13"; SHA is in a tooltip.

**"Backend update required"** becomes a **persistent banner at the top of the app**
when any installed backend needs updating, with a single "Update all" action. Not a
buried row note.

### Marketplace

**Rename and refocus.** Today's "Marketplace" is a directory of *other* apps that
talk to Lemonade. That's valuable but not the marquee position. Two changes:

1. **Marketplace becomes "Connect."** Smaller scope, clearer noun. Each card has a
   primary action that *actually does something*: "Configure for AnythingLLM" copies
   the OpenAI-compatible endpoint + key into the user's clipboard, deep-links to
   AnythingLLM's settings (via `openExternal`), and confirms. Visit-and-guide
   become secondary actions in a `⋯` menu.
2. **A new top-level "Discover" tab** surfaces *models*, *personas*, and *prompts* (the
   Open WebUI move) — pulled from a Lemonade-curated JSON manifest at build/release
   time, NOT from a runtime server endpoint (invariant #11). This is what users
   actually want a "marketplace" to be.

Card refresh: less Yahoo!, more App Store. Larger icon, single-line tagline,
primary CTA pill, secondary actions hidden behind `⋯`.

### Status bar

Default: single pill, bottom-left only: `● Connected · Qwen3-26B · 44 tok/s`. The
TPS is the most universally-useful single number. Click the pill → expands into the
full strip (Input/Output tokens, TTFT, RAM, CPU). Keyboard `⌘.` toggles. Pin state
in localStorage so power users keep it open.

### Settings

Promote from bottom-of-rail-gear to **a real top-level destination**, accessible
both from the rail bottom AND from a profile menu in the title bar (top-right).
Settings becomes the home for Backends, API keys, theme, zoom, telemetry,
about/version, and the new Memory feature. Use the macOS System Settings sidebar
pattern: list of sections on the left, detail on the right, search at the top.

### 2026-05-15T20:03:13Z: Recipes UI surface (v1.2) for prototype

**By:** Mattingly (UI/UX Designer), requested by Kyle

**Scope:** Added a v1.2 "Recipes" surface to `prototype/ui-redesign/` (static prototype only — NOT wired to lemond, NOT in src/cpp/, src/app/, or src/web-app/).

**Decisions made during the build:**

1. **Terminology — "recipe" collision is real.** In the C++ codebase, `recipe` refers to the backend engine id (`llamacpp`, `flm`, `ryzenai-llm`, `whispercpp`, `sd-cpp`, `vllm`). In the user-facing prototype, "Recipe" means the portable user preset (engine config + sampling). To avoid confusing users (and ourselves), the UI calls the C++ concept **engine** or **backend** everywhere copy mentions it — never "recipe." This is consistent in: the recipe slide-over's "Compatible engines" section, the model slide-over hint text ("This model uses the ryzenai-llm engine"), the recipe-card engine list, and tooltips on disabled compatibility options. Open question: do we want to rename the C++ concept later to reduce future translation cost? Probably not worth a churn, but flagging.

2. **Two-tier data model.** Recipe (portable, model-less) and AppliedRecipe (Recipe + model binding) are separate concepts. Recipes live in two tiers — Starters (bundled, read-only) and Yours (user-owned). Applied bindings are surfaced as a third zone "Applied to models." This split lets one recipe apply to many models and makes import/export trivial (a Recipe has no model dependencies).

3. **Recipe = `engine config` + `sampling`, surfaced as two sections.** The slide-over deliberately splits these because they have different lifecycles: `engine config` (ctx_size, backend, *_args) is applied at `/load`; `sampling` (temperature, top_p, top_k, repeat_penalty) is applied per `/chat/completions` request. The slide-over labels each section with that lifecycle hint inline. This surfaces lemond's actual contract to the user without making them read the API docs.

4. **Six bundled starters.** Balanced (default), Quality (long-form), Fast (snappy), Creative (high-temp), Long Context (32k ctx), Code (low-temp, tight sampling). Each declares compatible engines explicitly. Starter recipes are read-only — the slide-over disables controls and replaces "Save" with "Clone to edit."

5. **Compatibility filtering.** When applying a recipe to a model, incompatible models appear in the select but are `disabled` with a `title` tooltip explaining which engine the model uses and why the recipe isn't compatible. Same approach in the model slide-over's recipe selector. The chat popover uses `aria-disabled` (since it's a custom list) with the same tooltip. This is cheaper than hiding incompatible options and teaches the user the engine model.

6. **Chat composer pill + popover.** A small `Model · ◐ Recipe ▾` pill sits above the composer bar. Clicking it opens a popover with the current recipe summary + a switchable list of all recipes (incompatible ones disabled) + an "Edit values for this conversation only" affordance (currently a toast — implies a future per-conversation override).

7. **Drag-and-drop import is visual only.** The recipes view has a `.is-dropping` overlay that appears on `dragenter` and fires a toast on `drop`. No JSON parsing, no validation. Production would need a schema-preview modal before commit.

8. **Engine→model lookup is a JS map, not HTML attributes.** `MODEL_ENGINES` in `app.js` maps `data-name` → engine id. I chose not to edit every existing model row to add `data-engine` because the rows are dense and a JS lookup is just as good for a prototype. In production this should come from the model registry response.

9. **No new design tokens.** Reused everything from `tokens.css`. Specifically: `--accent`/`--accent-soft`/`--accent-strong` for highlight states, `--surface-1/2/3/glass` for layering, `--success`/`--warn`/`--danger` not needed (no status semantics in this feature yet). Verified by audit that no new color, motion, or layout token was required.
